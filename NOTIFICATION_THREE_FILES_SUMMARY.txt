PURPOSE:      Low-level Firebase Cloud Messaging wrapper
LOCATION:     backend/app/Services/FCMService.php
SIZE:         200 lines

KEY METHODS:
  ✓ sendToDevice(token, title, body, data)
  ✓ sendToDevices(tokens, title, body, data)
  ✓ sendToTopic(topic, title, body, data)
  ✓ sendToBranchStaff(branchId, title, body, data)

STRENGTHS:

  ✓ Handles Android + iOS properly

CRITICAL ISSUES:

  ✗ N+1 queries in sendToBranchStaff() - loops through staff inefficiently
  ✗ NO retry logic - failed sends are lost forever
  ✗ NO audit trail - can't track success/failure
  ✗ Database queries in service - violates Single Responsibility

EXAMPLE PROBLEM:
  $fcm->sendToBranchStaff($branchId, 'title', 'body');
  // This does:
  // 1. SELECT * FROM users WHERE branch_id=? AND role='staff' (N+1!)
  // 2. For each staff: sendToDevice() - another Firebase call
  // 3. If FCM fails on 5th staff, notification 1-4 succeeded but no way to know
  // 4. No retry - permanently lost

EXAMPLE PROBLEM #1: Silent FCM Failure
  public static function sendToCustomer(...) {
      // Create DB record (always succeeds)
      $notification = Notification::create([...]);  ✓ Success

      // Send FCM (might fail)
      try {
          $customer->fcm_token && FCMService::sendToDevice(...);  ✗ Might fail
      } catch (Exception $e) {
          Log::warning('FCM failed');  // Just log, don't retry
      }
      // Result: DB says notification created, but customer never got push!
      // No way to know it failed, no retry mechanism
  }

EXAMPLE PROBLEM #2: N+1 Queries
  public static function sendToBranchStaff($branchId, ...) {
      $staffUsers = User::where('branch_id', $branchId)  // Query 1
                        ->where('role', 'staff')
                        ->get();  // N results

      foreach ($staffUsers as $staff) {  // N iterations
          self::sendToUser(                // Creates 1 DB record per staff
              $staff->id, ...
          );  // N more queries (INSERT)
      }
      // For 10 staff: 1 SELECT + 10 INSERT = 11 queries
      // Should be: 1 SELECT + 1 batch INSERT = 2 queries (80% reduction!)
  }

THE SOLUTION (NotificationManager)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Unified Flow:

    LaundryController
         │
         └─→ NotificationManager::send('payment_approved', $laundry, $context)
                 │
                 ├─ Lookup event config (NotificationEvent table)
                 │
                 ├─ Render template with context
                 │
                 ├─ Create audit record (NotificationDelivery)
                 │
                 ├─ Send FCM with retry (up to 3 times)
                 │
                 ├─ IF success: Mark delivered, log success
                 │
                 ├─ IF fails: Update status, queue for retry
                 │
                 └─ Return delivery status

Benefits:
  ✓ Single entry point for all notifications
  ✓ Configuration-driven (NotificationEvent table)
  ✓ Complete audit trail (NotificationDelivery table)
  ✓ Automatic retry with backoff
  ✓ Consistent error handling
  ✓ Batch operations (no N+1 queries)
  ✓ Testable (dependency injection)
  ✓ Extensible (email, SMS, webhook support)

   • sendToDevice() exists in FCMService AND FirebaseNotificationService
   • FCM logic scattered across all 3 files
   • Business logic mixed with infrastructure code

3. ERROR HANDLING
   • Silent failures (FCM errors just logged)
   • No retry mechanism
   • No way to know if notification succeeded

4. AUDIT TRAIL
   • Only NotificationService creates DB records
   • But success/failure of FCM unknown
   • Can't troubleshoot delivery issues

5. PERFORMANCE
   • N+1 queries in sendToBranchStaff() (inefficient)
   • No batch operations
   • No connection pooling

6. TESTABILITY
   • All static methods in NotificationService (can't mock)
   • Heavy dependencies on database and Firebase
   • <10% test coverage
